¿Qué es la Activity y su ciclo de vida?

Una Activity es el punto de entrada para que el usuario interactúe con la app. Durante su existencia, pasa por una serie de estados — eso es su ciclo de vida.

¿Por qué importa?

Si no manejamos bien el ciclo de vida, la app puede crashear, perder datos del usuario, consumir batería en segundo plano o comportarse de forma impredecible.

Modern Android Development

Hoy una sola Activity puede mostrar múltiples pantallas intercambiándolas con Navigation Compose. Ya no es 1 Activity = 1 pantalla.

Los 7 métodos del ciclo de vida

Resumed
Activity tiene foco
Started
Activity es visible
Created
Activity en memoria
Destroyed / Initialized
onCreate()

Se ejecuta una sola vez al crear la Activity. Aquí se infla la UI y se inicializa el ViewModel.

onStart()

La Activity está a punto de hacerse visible. Aún no puede interactuar con el usuario.

onResume()

La Activity tiene el foco. El usuario puede interactuar. Aquí vive la mayoría del tiempo.

onPause()

Pierde el foco pero sigue visible (ej: un diálogo encima). Libera recursos que no necesita.

onStop()

La Activity ya no es visible. Sigue en memoria. Detener operaciones costosas aquí.

onDestroy()

La Activity es eliminada definitivamente. Última oportunidad para liberar recursos.

onRestart()

Se ejecuta cuando una Activity en onStop() vuelve a primer plano en vez de recrearse.

Bloque 1 · Ciclo de vida

Escenarios reales del ciclo de vida

Abrir la app

onCreate → onStart → onResume
La Activity sube hasta Resumed y espera interacción del usuario.

Rotar la pantalla

onPause → onStop → onDestroy → onCreate → ...
Android destruye y recrea la Activity. El estado se pierde si no lo guardamos.

App a segundo plano

onPause → onStop
El sistema puede matar el proceso si necesita memoria. Al volver: onRestart → onStart → onResume.

💡 La rotación de pantalla es el escenario más común donde se pierde el estado — aquí es donde entra rememberSaveable.

Registro con Log — depurando el ciclo de vida

La clase Log nos permite escribir mensajes en el Logcat de Android Studio para observar en tiempo real qué métodos se ejecutan.

class MainActivity : ComponentActivity() {

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        Log.d("MainActivity", "onCreate llamado")      // D = Debug
    }

    override fun onStart()   { super.onStart();   Log.d("MainActivity", "onStart llamado") }
    override fun onResume()  { super.onResume();  Log.d("MainActivity", "onResume llamado") }
    override fun onPause()   { super.onPause();   Log.d("MainActivity", "onPause llamado") }
    override fun onStop()    { super.onStop();    Log.d("MainActivity", "onStop llamado") }
    override fun onDestroy() { super.onDestroy(); Log.d("MainActivity", "onDestroy llamado") }
}
Log.d(tag, msg)

tag: identifica la clase (filtra en Logcat). msg: el mensaje que quieres ver. Usa D para debug.

Niveles de Log

Log.v Verbose · Log.d Debug · Log.i Info · Log.w Warning · Log.e Error

¿Qué es SavedInstanceState?

Es un objeto Bundle que Android usa para guardar y restaurar datos cuando la Activity se destruye y recrea — por rotación, cambio de configuración o porque el sistema necesita memoria.

¿Cuándo actúa?

Rotación de pantalla · App a segundo plano · Sistema mata el proceso · Cambio de idioma del sistema

¿Qué puede guardar?

Tipos primitivos y Parcelable. No objetos grandes. Es un salvavidas para estado de IU, no una base de datos.

Activity destruida
→
Bundle guardado
→
Activity recreada
→
Bundle restaurado
💡 En Compose, rememberSaveable usa SavedInstanceState internamente — no necesitas manejarlo a mano.

Composición y recomposición

La composición es la descripción de la IU que Compose construye al ejecutar tus funciones componibles. En la primera ejecución guarda ese árbol en un objeto Composition.

Datos de entrada
→
Funciones componibles
→
Composition (IU)
Composición inicial

Primera ejecución. Compose registra todos los componibles y construye el árbol de la IU.

Recomposición

Ante un cambio de estado, Compose re-ejecuta solo los componibles afectados y actualiza la IU.

💡 La Composition solo puede crearse una vez y modificarse únicamente a través de la recomposición.
Bloque 2 · Compose

Estado en Compose

El estado es cualquier valor de la app que puede cambiar con el tiempo. Determina qué muestra la IU en un momento determinado.

Ejemplos de estado

Mensajes recientes de un chat · Foto de perfil del usuario · Posición de scroll en una lista

Tipos en Compose

State<T> — inmutable, solo lectura.
MutableState<T> — mutable, permite escritura.

Usa mutableStateOf() para crear un estado observable. Compose hará seguimiento y programará recomposición cuando cambie.

Eventos en Compose

Si el estado describe qué se muestra, los eventos explican por qué cambia. Son entradas que disparan una actualización del estado.

Interacción del usuario — presionar un botón, escribir en un campo, hacer scroll.

Factores externos — respuestas de red, datos de sensores, temporizadores.

→

💡 El estado es. Los eventos suceden.

Composición

Árbol que Compose construye y mantiene. Solo se modifica por recomposición.

Estado

Valor que cambia en el tiempo. Se declara con mutableStateOf() para ser observado.

Eventos

Acciones del usuario o sistema que modifican el estado y desencadenan recomposición.

Recomposición

Re-ejecución automática y eficiente de solo los componibles afectados.

💡 UDF (Unidirectional Data Flow): los eventos fluyen hacia arriba, el estado fluye hacia abajo.

Evolución del estado en 4 iteraciones

Cada iteración soluciona un problema concreto de la anterior

Bloque 3 · Iteración 1

val amountInput = "0"

Una variable val normal. Funciona para datos estáticos, pero Compose no la observa — nunca sabrá que cambió.

@Composable
fun TipCalculator() {
    val amountInput = "0"          // ❌ val: inmutable, Compose no la observa
    val amount = amountInput.toDoubleOrNull() ?: 0.0
    val tip = calculateTip(amount)

    TextField(
        value = amountInput,
        onValueChange = { /* amountInput = it  ← error de compilación */ }
    )
}
❌ Problema

No se puede reasignar. El campo de texto nunca actualiza su valor. La IU queda congelada.

¿Qué necesitamos?

Un valor que pueda cambiar y que Compose sepa que debe recomponer cuando cambie.

Bloque 3 · Iteración 2

var amountInput = mutableStateOf("0")

Ahora el valor es observable por Compose. Cuando cambia, programa una recomposición. Pero hay un problema oculto.

@Composable
fun TipCalculator() {
    var amountInput = mutableStateOf("0")  // ⚠️ se recrea en cada recomposición
    val amount = amountInput.value.toDoubleOrNull() ?: 0.0

    TextField(
        value = amountInput.value,
        onValueChange = { amountInput.value = it }   // dispara recomposición
    )
}
❌ Problema

Cada recomposición recrea el objeto desde cero. El valor escrito por el usuario se pierde en cada recomposición.

¿Qué necesitamos?

Que el estado sobreviva a la recomposición y Compose recuerde su valor entre ejecuciones.

Bloque 3 · Iteración 3

var amountInput by remember { mutableStateOf("") }

remember le dice a Compose que guarde el valor en la Composition y lo reutilice en vez de recrearlo en cada recomposición.

@Composable
fun TipCalculator() {
    var amountInput by remember { mutableStateOf("") }  // ✅ sobrevive a recomposición
    val amount = amountInput.toDoubleOrNull() ?: 0.0
    val tip = calculateTip(amount)

    TextField(
        value = amountInput,
        onValueChange = { amountInput = it }   // sintaxis limpia con delegación 'by'
    )
}
✅ Resuelto

El valor persiste entre recomposiciones. La delegación by evita escribir .value en cada acceso.

❌ Problema restante

Si el sistema rota la pantalla o mata el proceso, el estado se pierde — solo vive en la Composition.

Bloque 3 · Iteración 4

var amountInput by rememberSaveable { mutableStateOf("") }

rememberSaveable guarda el estado en el Bundle de Android (SavedInstanceState), sobreviviendo a rotaciones y a la muerte del proceso.

@Composable
fun TipCalculator() {
    var amountInput by rememberSaveable { mutableStateOf("") }  // ✅ solución completa
    val amount = amountInput.toDoubleOrNull() ?: 0.0
    val tip = calculateTip(amount)

    TextField(
        value = amountInput,
        onValueChange = { amountInput = it }
    )
}
✅ Sobrevive a recomposición

Igual que remember — el valor persiste entre recomposiciones.

✅ Sobrevive a rotación

Se guarda en el Bundle. Rotar la pantalla no borra lo que el usuario escribió.

💡 Regla práctica: usa rememberSaveable por defecto para estado de IU. Usa remember solo para objetos que no necesitan persistir entre cambios de configuración.
