¿Qué es un Code Review?

La práctica más importante del desarrollo profesional

Code Review · Definición

¿Qué es y para qué sirve?

Un code review es el proceso en el que uno o más developers leen y analizan el código de un compañero antes de que sea integrado al proyecto. No es para buscar errores — es para mejorar colectivamente.

Encontrar bugs temprano

Un error detectado en code review cuesta 10x menos que uno detectado en producción.

Compartir conocimiento

El que revisa aprende del código. El que es revisado aprende de los comentarios.

Mantener estándares

Asegura que todo el equipo escriba código con el mismo estilo y criterio.

💡 En empresas como Google, ningún código llega a producción sin pasar por al menos un code review. Es una habilidad tan importante como saber programar.
Code Review · Cómo hacerlo bien

Reglas de un buen code review

❌ Lo que NO se hace

🚫

Criticar a la persona — se revisa el código, no al developer

🚫

Decir "esto está mal" sin explicar por qué ni cómo mejorarlo

🚫

Ignorar lo que está bien — el feedback positivo también cuenta

🚫

Cambiar cosas por gusto personal sin razón técnica

🚫

Hacer el review de afán sin leer el código con calma

✅ Lo que SÍ se hace

✔

Preguntar antes de asumir: "¿Por qué decidiste hacerlo así?"

✔

Sugerir, no imponer: "Podrías considerar..."

✔

Reconocer explícitamente lo que está bien hecho

✔

Explicar el impacto: "Esto puede causar un bug cuando..."

✔

Ser específico: señalar la línea exacta, no el archivo completo

💡 Hoy en parejas: el objetivo es aprender juntos. Busquen mejoras — no busquen ganar.

Nombres y legibilidad

El código se lee más veces de las que se escribe

Legibilidad · Nombres de variables

Nombres que se explican solos

❌ Difícil de leer
var x by remember { mutableStateOf(0) }
var flag by remember { mutableStateOf(false) }
var s by remember { mutableStateOf(null) }
var f by remember { mutableStateOf(false) }

// ¿Qué hace esto?
if (flag && s != null) {
    x++
    f = true
}
✅ Auto-documentado
var currentIndex by remember { mutableStateOf(0) }
var isRevealed by remember { mutableStateOf(false) }
var selectedAnswer by remember { mutableStateOf(null) }
var isFinished by remember { mutableStateOf(false) }

// Ahora se entiende sin comentarios
if (isRevealed && selectedAnswer != null) {
    score++
    isFinished = true
}
💡 Regla de oro: si necesitas un comentario para explicar qué hace una variable, el nombre está mal. Un buen nombre es su propio comentario.

isFinished · isRevealed · hasError

Estado y lógica

Los errores más comunes del Taller 1

Estado · Reset incompleto

Error #1 — El estado que no se limpia

Al avanzar de pregunta algunos estados no se reseteaban — la respuesta de la pregunta anterior seguía visible en la siguiente.

❌ Reset incompleto
Button(onClick = {
    currentIndex++
    // selectedAnswer sigue con valor anterior
    // showFunFact sigue en true
    // la siguiente pregunta muestra
    // la respuesta anterior
}) {
    Text("Siguiente")
}
✅ Reset completo
Button(onClick = {
    currentIndex++
    selectedAnswer = null   // limpia selección
    showFunFact    = false  // oculta funFact

    if (currentIndex >= quizQuestions.size) {
        isFinished = true
    }
}) {
    Text("Siguiente")
}
💡 Cada vez que el estado cambia de contexto (nueva pregunta, nueva pantalla) pregúntate: ¿qué variables necesito resetear?
Estado · Lógica en la UI

Error #2 — Lógica de negocio dentro del onClick

❌ Todo mezclado en la UI
Button(onClick = {
    selectedAnswer = option
    if (option == currentQuestion.correctAnswer) {
        score++
    }
    showFunFact = true
    if (currentIndex + 1 >= quizQuestions.size) {
        isFinished = true
    }
}) {
    Text(option)
}
✅ Lógica extraída a una función
fun onOptionSelected(selected: String) {
    selectedAnswer = selected
    if (selected == currentQuestion.correctAnswer) {
        score++
    }
    showFunFact = true
    if (currentIndex + 1 >= quizQuestions.size) {
        isFinished = true
    }
}

// En el composable, limpio:
Button(onClick = { onOptionSelected(option) }) {
    Text(option)
}
💡 El onClick debería decir qué pasa, no cómo pasa. Más de 2 líneas → extraer a una función.
Estado · Condicionales

Error #3 — if anidados sin claridad

❌ if anidados difíciles de seguir
if (!isFinished) {
    if (selectedAnswer != null) {
        if (showFunFact) {
            Text(currentQuestion.funFact)
            if (currentIndex < quizQuestions.size - 1) {
                Button(onClick={}) { Text("Siguiente") }
            } else {
                Button(onClick={}) { Text("Ver resultado") }
            }
        }
    }
} else { /* pantalla de resultado */ }
✅ when con composables separados
when {
    isFinished -> {
        ResultScreen(
            score = score,
            onRestart = ::restart
        )
    }
    selectedAnswer != null -> {
        AnswerSection(
            question = currentQuestion,
            onNext = ::goToNext
        )
    }
    else -> {
        OptionsSection(
            options = currentQuestion.options,
            onSelect = { selectedAnswer = it }
        )
    }
}
💡 Si tu if tiene más de 2 niveles de anidación, usa when o extrae composables separados.

Composables reutilizables

No repitas UI — extrae componentes

Composables · Extracción

Identificar y extraer composables

❌ Botón de opción repetido 4 veces
// opción A
Button(
    onClick = { selectedAnswer = "Opción A" },
    modifier = Modifier.fillMaxWidth()
) { Text("Opción A") }

// opción B — código idéntico
Button(
    onClick = { selectedAnswer = "Opción B" },
    modifier = Modifier.fillMaxWidth()
) { Text("Opción B") }

// ... y así para C y D
✅ Un composable + forEach
@Composable
fun OptionButton(
    text: String,
    selectedAnswer: String?,
    correctAnswer: String,
    onClick: () -> Unit
) {
    Button(
        onClick = onClick,
        enabled = selectedAnswer == null,
        modifier = Modifier.fillMaxWidth()
    ) { Text(text) }
}

// Uso limpio
currentQuestion.options.forEach { option ->
    OptionButton(
        text = option,
        selectedAnswer = selectedAnswer,
        correctAnswer = currentQuestion.correctAnswer,
        onClick = { selectedAnswer = option }
    )
}
Composables · Cuándo extraer

¿Cuándo extraer un composable?

→

El mismo bloque de UI aparece más de una vez

→

Un bloque tiene una responsabilidad clara y propia

→

El composable padre supera las ~80 líneas

→

Quieres reutilizarlo en otra pantalla

💡 Un composable debe hacer una sola cosa. Si su nombre necesita un "y", probablemente son dos composables.
// Composables bien separados
// para el Taller 1

@Composable
fun QuizScreen() { /* orquesta todo */ }

@Composable
fun QuestionCard(question: String) { }

@Composable
fun OptionButton(...) { }

@Composable
fun FunFactSection(text: String) { }

@Composable
fun ScoreIndicator(score: Int, total: Int) { }

@Composable
fun ResultScreen(
    score: Int,
    onRestart: () -> Unit
) { }

Resumen

Lo que revisamos hoy

Code Review profesional

Revisar código es una habilidad. Se comenta el código, no la persona. Se sugiere, no se impone.

Nombres y convenciones

camelCase variables, PascalCase clases. Prefijo is/has para booleanos. Sin abreviaciones.

Reset del estado

Al cambiar de pregunta resetear todos los estados relacionados. Un reset incompleto causa bugs invisibles.

Lógica fuera de la UI

El onClick no debe tener lógica compleja. Extraer funciones con nombres que describan qué hacen.

Composables reutilizables

Si un bloque de UI se repite, se extrae. Un composable = una responsabilidad. Usar forEach en vez de repetir.

Estructura de archivos

Separar por responsabilidad: model/, data/, ui/screens/, ui/components/.
