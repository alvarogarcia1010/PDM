El problema

Vean este código

Una pantalla que muestra una lista de películas. Datos hardcoded, todo en un solo Composable.

@Composable
fun MovieListScreen() {
    var movies by remember { mutableStateOf(listOf<Movie>()) }
    var loading by remember { mutableStateOf(false) }

    LaunchedEffect(Unit) {
        loading = true
        delay(2000)
        movies = listOf(
            Movie("Minecraft", 9.0),
            Movie("Dune", 8.7),
            Movie("Interstellar", 9.3)
        )
        loading = false
    }

    if (loading) CircularProgressIndicator()
    else LazyColumn {
        items(movies) { movie ->
            Text("${movie.title} - ${movie.rating}")
        }
    }
}
El problema

¿Qué pasa cuando la app crece?

El código de arriba funciona para 3 películas hardcoded. Pero pensemos en una app real.

!

Los datos no se pueden compartir. Si otra pantalla necesita las películas, las tiene que cargar de nuevo desde cero.

!

La lógica está atrapada en la UI. No la podemos testear sin renderizar el composable completo.

!

Si cambia la fuente de datos (de hardcoded a API), tenemos que tocar el composable.

!

Al rotar la pantalla, el estado se pierde y los datos se vuelven a cargar.

💡 Esto se vuelve un infierno cuando la app tiene 20 pantallas. Necesitamos una forma de organizar el código.
Solución

¿Qué es una arquitectura de software?

Una arquitectura no es más que una forma de organizar el código para poder mantener el control sobre la aplicación a medida que crece.

En apps móviles esto es esencial para gestionar tres cosas:

→
El ciclo de vida — qué pasa cuando la pantalla se destruye y se vuelve a crear.
→
La interfaz de usuario — qué se muestra y cómo se actualiza.
→
La lógica de negocio — qué hace la app y cómo lo hace.
💡 Una buena arquitectura define qué responsabilidad tiene cada parte del código y dónde vive cada cosa.
Solución

Arquitectura recomendada por Google

Teniendo en cuenta los principios de arquitectura comunes, diseña cada aplicación con al menos dos capas:

UI Layermuestra los datos en la pantalla
↓
Data Layercontiene la lógica y expone los datos

Puedes agregar una capa adicional llamada capa de dominio para simplificar y volver a utilizar las interacciones entre la UI y la capa de datos.

Domain Layeropcional
💡 En este curso vamos a trabajar solo con UI Layer y Data Layer. La capa de dominio aparece en proyectos grandes con lógica compleja reutilizable.
Capa de UI

Capa de la UI

La función de la capa de UI (o capa de presentación) consiste en mostrar los datos de la aplicación en la pantalla.

Cuando los datos cambian — ya sea por una interacción del usuario o por una entrada externa como una respuesta de red — la UI se actualiza para reflejar los cambios.

UI Layer
UI elements
↓
State holders
💡 La capa de UI consta de dos cosas: elementos de UI que renderizan los datos (Composables) y contenedores de estado como el ViewModel que retienen los datos y exponen la lógica.
Patrón de arquitectura

¿Cómo organizamos esa capa por dentro?

Acabamos de ver que la capa de UI tiene UI elements y State holders. Pero, ¿cómo se relacionan entre ellos? ¿Quién le habla a quién? ¿Quién maneja la lógica?

Para responder eso, necesitamos hablar de patrones de arquitectura.

💡 Una arquitectura es el plano completo de la app — las capas que acabamos de ver. Un patrón de arquitectura es la receta específica para organizar una de esas capas. MVC, MVP y MVVM son patrones para la capa de UI.
Patrón de arquitectura

Las arquitecturas que vinieron antes

Antes de MVVM hubo otras arquitecturas. Cada una resolvió problemas de la anterior.

MVC
Model — View — Controller. La primera. Vista y controlador viven en la misma Activity.
✗ Una sola clase con todo
MVP
Model — View — Presenter. Saca la lógica del Activity. Mejora testing.
✓ Lógica separada
✗ Acoplado a la vista
MVVM
Model — View — ViewModel. La vista observa datos del ViewModel y se actualiza sola.
✓ Reactivo y testeable
💡 MVVM ganó porque la vista no le pregunta al ViewModel qué hacer — observa cambios y se actualiza automáticamente.

MVVM

MVVM en detalle

La arquitectura MVVM divide la responsabilidad en tres piezas claras:

Model
Los datos. Las clases que representan la información que la app maneja. Por ejemplo: data class Movie(...)
View
Lo que el usuario ve. En Compose, son los Composables. La View no decide nada por sí misma — observa al ViewModel y se actualiza cuando los datos cambian.
ViewModel
El cerebro intermedio. Conecta los datos con la vista. Maneja el estado, la lógica, y expone los datos como flujos observables.
💡 Los cambios en el ViewModel actualizan la vista automáticamente, sin que la vista tenga que pedirlos.

MVVM

Estructura típica de carpetas

Cada pantalla tiene su propia carpeta con su View y ViewModel juntos. El modelo y los datos viven aparte.

com.example.app
├── data
│   └── DummyMovies.kt
├── model
│   └── Movie.kt
└── screens
    └── MovieList
        ├── MovieListScreen.kt
        └── MovieListViewModel.kt
💡 Cuando abrís un proyecto MVVM, sabés inmediatamente dónde buscar cada cosa. Eso es lo que vale la arquitectura.
Implementación

El Modelo

El Modelo es lo más simple: una clase de datos que representa la información de la app.

// model/Movie.kt
data class Movie(
    val title: String,
    val rating: Double
)
💡 El modelo no sabe nada de la UI ni de cómo se muestran los datos. Solo describe la información.
Implementación

Datos de ejemplo

Mientras no consumimos una API, podemos usar datos hardcoded en un archivo aparte:

// data/DummyMovies.kt
val dummyMovies = listOf(
    Movie("Minecraft", 9.0),
    Movie("Dune", 8.7),
    Movie("Interstellar", 9.3),
    Movie("Inception", 8.8),
    Movie("The Matrix", 8.7)
)
💡 Tener los datos en un archivo aparte facilita reemplazarlos después por una API real sin tocar el ViewModel ni la View.
Implementación

¿Cómo fluyen los datos en MVVM?

Antes de meternos en el ViewModel, hay un concepto clave: en MVVM la información tiene una sola dirección.

ViewModel
estado
↓
lista de películas
click del usuario
↑
eventos
View
💡 Esto se llama flujo unidireccional de datos. El estado fluye hacia abajo, los eventos fluyen hacia arriba. Para que esto funcione, necesitamos una variable especial que la View pueda observar.
Implementación

StateFlow — el puente reactivo

Esa variable especial se llama StateFlow. Es parte de Kotlin Coroutines.

StateFlow es una variable que siempre tiene un valor y notifica a quien la observe cuando cambia.

🎬 Pensalo así: en YouTube, te suscribís a un canal y cuando sube un video nuevo te llega la notificación automáticamente. No tenés que ir a revisar todos los días. StateFlow funciona igual: la View se "suscribe" al ViewModel y recibe avisos cuando los datos cambian.
✓
Siempre tiene un valor inicial — ideal para representar estado.
✓
Compose se recompone automáticamente cuando el valor cambia.
✓
Es seguro para múltiples hilos y trabaja bien con corrutinas.
Implementación

Mutable vs Inmutable

StateFlow tiene dos versiones, y se usan juntas con un patrón muy específico:

privado
MutableStateFlow — la versión modificable. Solo el ViewModel la puede cambiar. Por convención se le pone guion bajo: _movies
público
StateFlow — la versión solo lectura. Lo que la View puede observar pero no modificar. Sin guion bajo: movies
private val _movies = MutableStateFlow<List<Movie>>(dummyMovies)
val movies = _movies.asStateFlow()
💡 Esto se llama patrón de encapsulamiento. Garantiza que solo el ViewModel pueda cambiar el estado. La View solo lo observa. asStateFlow() hace la conversión de mutable a inmutable.
Implementación

El ViewModel

Ahora que entendemos StateFlow, el ViewModel se vuelve simple. Mantiene el estado y lo expone a la View.

// screens/MovieList/MovieListViewModel.kt
class MovieListViewModel : ViewModel() {

    private val _movies = MutableStateFlow<List<Movie>>(dummyMovies)
    val movies = _movies.asStateFlow()
}
💡 Eso es todo el ViewModel por ahora. Tiene el estado privado, lo expone como inmutable, y los datos están listos. Es el patrón completo aplicado.
Implementación

La View

La View observa el StateFlow del ViewModel con collectAsState y se recompone cuando los datos cambian.

// screens/MovieList/MovieListScreen.kt
@Composable
fun MovieListScreen(
    viewModel: MovieListViewModel = viewModel()
) {
    val movies by viewModel.movies.collectAsState()

    LazyColumn {
        items(movies) { movie ->
            Text("${movie.title} - ${movie.rating}")
        }
    }
}
💡 La View no carga datos, no decide nada. Solo observa al ViewModel y muestra lo que ve. Si los datos cambian, Compose se recompone automáticamente.
Cierre

¿Qué ganamos con MVVM?

Volvamos al código del inicio. Compará lo que teníamos con lo que tenemos ahora.

✓
El ViewModel sobrevive a la rotación — los datos no se pierden cuando el usuario gira la pantalla.
✓
La lógica está separada de la UI — podemos testear el ViewModel sin renderizar nada.
✓
Los datos son reactivos — la UI se actualiza sola cuando cambian.
✓
Sabemos dónde buscar cada cosa — model, screens, viewmodel.
✓
Es lo que la industria usa — 76% de los proyectos Android en producción.
💡 El código no es más corto. Pero es infinitamente más mantenible cuando la app crece.
