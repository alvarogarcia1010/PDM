Dependencias

Repositorios Maven y Version Catalog

Setup · Repositorios Maven

¿De dónde vienen las librerías?

Cuando agregamos una dependencia, Android Studio va a buscarla a internet — a un repositorio Maven. El proyecto tiene dos configurados por defecto en settings.gradle.kts.

google()

Repositorio oficial de Google. Aquí vive todo Jetpack: Compose, Navigation, Room, Hilt…

mavenCentral()

Repositorio público de la comunidad. Aquí vive Retrofit, Moshi, librerías de terceros…

// settings.gradle.kts
dependencyResolutionManagement {
    repositories {
        google()        // ← todo Jetpack
        mavenCentral()  // ← comunidad
    }
}
💡 Piensen en los repositorios como las "tiendas" de las librerías. google() y mavenCentral() cubren casi todo lo que van a necesitar en el curso.
Setup · Version Catalog

Un solo archivo para todas las versiones

Antes, cada versión se escribía directo en build.gradle. Si la misma versión se usaba en varios módulos, había que cambiarla en cada uno. Version Catalog lo centraliza en gradle/libs.versions.toml.

[versions]

Solo los números, con un alias para cada uno. Ej: navigation3 = "1.1.0"

[libraries]

El coordenado completo de cada librería, referenciando la versión por alias con version.ref.

[plugins]

Los plugins de Gradle: Android, Kotlin, Serialization. Se declaran aquí y se aplican en build.gradle.kts.

💡 Un solo archivo para todas las versiones. Cambiar una versión es cambiar una línea — y se actualiza en todo el proyecto automáticamente.
Setup · libs.versions.toml

Agregando Navigation 3 — libs.versions.toml

Solo necesitan agregar estas líneas al archivo existente. Los guiones (-) son la convención del TOML para separar palabras.

[versions]
+ navigation3              = "1.1.0"
+ kotlinxSerializationCore = "1.11.0"

[libraries]
+ androidx-navigation3-runtime = { group = "androidx.navigation3", name = "navigation3-runtime", version.ref = "navigation3" }
+ androidx-navigation3-ui      = { group = "androidx.navigation3", name = "navigation3-ui",      version.ref = "navigation3" }
+ kotlinx-serialization-core   = { group = "org.jetbrains.kotlinx", name = "kotlinx-serialization-core", version.ref = "kotlinxSerializationCore" }

[plugins]
+ kotlin-serialization = { id = "org.jetbrains.kotlin.plugin.serialization", version.ref = "kotlin" }
💡 Los guiones del TOML se convierten en puntos al referenciar: navigation3-runtime → libs.androidx.navigation3.runtime
Setup · build.gradle.kts

Agregando Navigation 3 — build.gradle.kts

Un plugin nuevo en plugins { } y tres dependencias en dependencies { }. El resto del archivo queda igual.

plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.kotlin.android)
    alias(libs.plugins.kotlin.compose)
+   alias(libs.plugins.kotlin.serialization)
}

dependencies {
    // ... las que ya tienen

+   // Navigation 3
+   implementation(libs.androidx.navigation3.runtime)
+   implementation(libs.androidx.navigation3.ui)
+   implementation(libs.kotlinx.serialization.core)

    implementation(libs.androidx.core.ktx)
    // ...
}
💡 Después de agregar estas líneas → Sync Project with Gradle Files. Android Studio descarga las librerías automáticamente.

La pila de navegación

El concepto que explica todo lo demás

La pila · Back Stack

Last In, First Out

La navegación en Android funciona con una pila — el último en entrar es el primero en salir. En Nav 3 esa pila no es una caja negra: es una lista de Kotlin que vos controlás.

Abro la app
Home
→
Voy a Profile
MyProfile
Home
→
Presiono atrás
Home
💡 Para avanzar se agrega un elemento. Para volver se quita el último. La pantalla visible siempre es la del tope.
💡 Nav 3 no es una evolución de Nav 2 — es una reescritura. El developer posee la pila en lugar de pedírsela a una caja negra.
// avanzar — push
backStack.add(Routes.MyProfile(id))

// volver — pop
backStack.removeLastOrNull()

Creando el backStack

Una lista observable de Kotlin

BackStack · mutableStateListOf

La pila como lista observable

val backStack = remember {
    mutableStateListOf<Any>(Routes.Home)
}

// avanzar
backStack.add(Routes.MyProfile("123"))

// volver
backStack.removeLastOrNull()
💡 mutableStateListOf es como mutableListOf pero Compose la observa — cuando cambia, la UI se recompone sola.

NavDisplay

Renderizando la pila — primer intento

NavDisplay · Buena práctica primero

Las pantallas no navegan — piden navegar

❌ Pasar el backStack completo
@Composable
fun HomeScreen(
    backStack: NavBackStack // ← conoce demasiado
) {
    Button(onClick = {
        backStack.add(
            Routes.MyProfile("123")
        )
    }) { Text("Ver perfil") }
}
✅ Pasar una lambda específica
@Composable
fun HomeScreen(
    navigateToProfile: (String) -> Unit
) {
    Button(onClick = {
        navigateToProfile("123")
    }) { Text("Ver perfil") }
}
💡 HomeScreen no sabe que existe un backStack — solo declara su intención. Es el mismo principio de UDF que ya conocen: los eventos suben, el estado baja.
NavDisplay · Primer intento

Conectando el backStack con la UI

NavDisplay lee el tope del backStack y renderiza la pantalla correspondiente. Por ahora usamos when para mapear cada ruta a su composable.

@Composable
fun NavigationWrapper() {
    val backStack = remember { mutableStateListOf<Any>(Routes.Home) }

    NavDisplay(
        backStack = backStack,
        onBack    = { backStack.removeLastOrNull() },
        entryProvider = { key ->
            when (key) {
                is Routes.Home ->
                    NavEntry(key) {
                        HomeScreen(navigateToProfile = { id ->
                            backStack.add(Routes.MyProfile(id))
                        })
                    }
                is Routes.MyProfile ->
                    NavEntry(key) {
                        MyProfileScreen(onBack = { backStack.removeLastOrNull() })
                    }
                else -> NavEntry(key = Unit) { Text("Error") }
            }
        }
    )
}
NavDisplay · ¿Qué huele mal?

Funciona… pero hay tres problemas

!

mutableStateListOf<Any> — la pila acepta cualquier objeto. El compilador no nos protege si agregamos algo que no es una ruta.

!

El when con is no es exhaustivo — si agregamos una ruta nueva y olvidamos el when, el compilador no dice nada.

!

El else con "Error" nunca debería existir si tenemos todas las rutas — es ruido que el compilador nos obliga a poner.

💡 Necesitamos un sistema de rutas que el compilador entienda. Así when se vuelve exhaustivo, desaparece el else, y la pila es type-safe.
5

Rutas type-safe

sealed class · NavKey · @Serializable · rememberNavBackStack

Rutas · El problema con <Any>

La pila acepta cualquier cosa… ¿eso está bien?

Con mutableStateListOf<Any> la pila acepta cualquier objeto. Funciona, pero el compilador no nos protege — si agregamos algo incorrecto, no lo sabremos hasta que la app explote en runtime.

val backStack = remember {
    mutableStateListOf<Any>(Routes.Home)
}

// El compilador acepta todo sin quejarse:
backStack.add("pantalla_detalle") // ← string
backStack.add(42)                 // ← número
backStack.add(Routes.MyProfile()) // ← ruta real

// ¿Cómo sé cuál es cuál?
// Solo en runtime — demasiado tarde
Necesitamos tres cosas

1. Que el compilador conozca todas las rutas posibles.
2. Que cada ruta pueda llevar sus propios argumentos.
3. Que Nav 3 pueda restaurar la pila si el sistema mata el proceso.

💡 La solución: sealed class + NavKey + @Serializable. Tres decisiones, tres problemas resueltos.
Rutas · data object vs data class

¿Cuándo usar cada uno?

Antes de definir las rutas, necesitamos entender la diferencia. La regla es simple: si el elemento tiene datos propios, usamos data class. Si no, data object.

data object — sin argumentos

Una sola instancia, siempre igual. Ideal para pantallas que no necesitan recibir información. Tiene toString() y equals() automáticos.

// ✅ Home no necesita argumentos
data object Home : Routes()

// Siempre es la misma instancia
Routes.Home == Routes.Home // true
data class — con argumentos

Cada instancia lleva sus propios datos. Ideal para pantallas de detalle que necesitan saber qué mostrar.

// ✅ MyProfile necesita saber qué perfil
data class MyProfile(val id: String) : Routes()

// Cada instancia es distinta
MyProfile("1") == MyProfile("2") // false
MyProfile("1") == MyProfile("1") // true
💡 La regla: ¿la pantalla necesita recibir información para saber qué mostrar? → data class. ¿Siempre muestra lo mismo? → data object.
Rutas · sealed class : NavKey

Rutas como objetos type-safe

sealed class Routes : NavKey {

    @Serializable
    data object Home : Routes()

    @Serializable
    data class MyProfile(
        val id: String
    ) : Routes()

}
sealed class

El compilador conoce todas las subclases posibles. Si olvidamos manejar una ruta en el when, nos avisa en compilación.

: NavKey

Requisito de Nav 3. Solo objetos que implementen NavKey pueden vivir en el backStack — adiós al <Any>.

Rutas · @Serializable

¿Por qué @Serializable?

Cuando Android necesita memoria, puede matar tu app en segundo plano. Cuando el usuario regresa, el sistema intenta reconstruirla — y Nav 3 necesita restaurar la pila completa.

Sin @Serializable

Nav 3 no puede guardar la ruta. El usuario pierde su lugar al volver a la app.

Con @Serializable

Nav 3 convierte la ruta y sus argumentos a JSON. Al volver, reconstruye el backStack exacto.

Por eso el plugin

El plugin kotlin.serialization que agregamos en el TOML es el que procesa @Serializable en compilación.

💡 Todo se conecta: plugin en libs.versions.toml → habilita @Serializable → Nav 3 puede guardar y restaurar la pila completa.
Rutas · rememberNavBackStack

De mutableStateListOf a rememberNavBackStack

Ahora que nuestras rutas implementan NavKey, Nav 3 nos da un helper que hace lo mismo — pero además persiste la pila ante rotaciones y muerte del proceso.

Antes
val backStack = remember {
    mutableStateListOf<Any>(Routes.Home)
}

// ❌ Tipo Any — no type-safe
// ❌ No persiste si el sistema
//    mata el proceso
Ahora
val backStack =
    rememberNavBackStack(Routes.Home)

// ✅ Tipo NavKey — type-safe
// ✅ Persiste automáticamente
//    (usa rememberSaveable por dentro)
💡 rememberNavBackStack usa rememberSaveable por dentro — el mismo concepto que ya vieron. La pila sobrevive rotaciones y process death.

NavDisplay y entryProvider

El composable que renderiza la pila

NavDisplay · ¿Qué es un DSL?

Ya usan DSLs — sin saberlo

Un DSL (Domain-Specific Language) es una API diseñada para leerse como un lenguaje propio. En lugar de llamar funciones una por una, declarás lo que querés dentro de un bloque estructurado.

// Scaffold — DSL que ya usan
Scaffold(
    topBar    = { TopAppBar(...) },
    bottomBar = { NavigationBar(...) },
    floatingActionButton = { FAB(...) }
) { innerPadding ->
    // contenido
}
// entryProvider — DSL de Nav 3
entryProvider {
    entry<Routes.Home> {
        HomeScreen(...)
    }
    entry<Routes.MyProfile> { route ->
        MyProfileScreen(...)
    }
}
💡 Llevan semanas usando DSLs de Compose. entryProvider { } sigue exactamente el mismo patrón — declarás las entradas dentro de un bloque y Nav 3 sabe cómo interpretarlas.
NavDisplay · El wrapper completo

Conectando todo con entryProvider DSL

@Composable
fun NavigationWrapper() {
    val backStack = rememberNavBackStack(Routes.Home)

    NavDisplay(
        backStack = backStack,
        onBack    = { backStack.removeLastOrNull() },
        entryProvider = entryProvider {

            entry<Routes.Home> {
                HomeScreen(
                    navigateToProfile = { id ->
                        backStack.add(Routes.MyProfile(id))
                    }
                )
            }

            entry<Routes.MyProfile> { route ->
                MyProfileScreen(
                    userId = route.id,
                    onBack = { backStack.removeLastOrNull() }
                )
            }

        }
    )
}
💡 entry<Routes.MyProfile> { route -> } — el parámetro route ya es el objeto deserializado. Accedemos a route.id directamente, sin cast, sin strings, sin else.
Resumen

Lo que construimos hoy

Version Catalog

Un solo archivo libs.versions.toml centraliza versiones. Cambiar una versión es cambiar una línea.

La pila

add() para avanzar, removeLastOrNull() para volver. El developer la controla directamente.

rememberNavBackStack

Persiste la pila automáticamente usando rememberSaveable. Sobrevive rotaciones y process death.

sealed class : NavKey + @Serializable

Rutas type-safe que el compilador verifica. @Serializable permite restaurar la pila.

entryProvider DSL

entry<Ruta> { route -> } — acceso type-safe a los argumentos, sin when, sin casos imposibles.

Lambdas de navegación

Las pantallas declaran su intención, no conocen el backStack. Más testeable, más reutilizable.
